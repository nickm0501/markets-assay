Description & Problem Statement

Build a local, rerunnable research pipeline that determines whether historical news sentiment contains useful out-of-sample information about subsequent U.S. equity and ETF returns, without live trading, on a zero vendor budget, using a frozen two-year free-data window (one development year, one untouched holdout year).

A researcher cannot determine whether news sentiment has tradable predictive value for U.S. equities/ETFs because no rerunnable, leakage-controlled, cost-aware pipeline exists to test that cheaply, within a zero-dollar vendor budget and a two-year free-data history ceiling.

Scope

In scope

  • Fixture-first → curated-sample → narrow free-source probe → full two-year hypothesis test (Stages 0–3).
  • Deterministic local sentiment scoring, symmetric long/short/flat rules, immutable dataset/observation lineage, decision-oriented reports.

Out of scope (V1 non-goals)

  • Live trading or brokerage execution, interactive dashboards, tick-level data, futures/options/exact benchmark-index feeds, portfolio optimization, supervised/deep-learning models, mandatory social-media ingestion, production-grade proof across many market regimes.

Current Direction

Staged delivery, per the spec: Stage 0 (deterministic fixture pipeline) → Stage 1 (hand-curated real sample) → Stage 2 (narrow Massive/GDELT/Alpaca probe) → Stage 3 (free two-year hypothesis test) → Stage 4 (paid history/modeling only if Stage 3 justifies it). Stage 0 is planned in full; Stages 1–4 are named but not yet planned in detail.

Round 1 — manual audit · 2026-07-09
2026-07-09Accepted
analysis.rs, backtest.rs, and report.rs must operate per configuration (news_window × measurement_horizon × source_set) — never pooled.
Rationale
The spec defines run_id as one configuration's result, and the plan states later stages reuse this loop unchanged — pooling here would defeat the First Experiment Matrix's purpose of comparing configurations.
Consequences
Reworked Tasks 7–9: per-configuration thresholds and trade streams, one recommendation per configuration instead of one blended verdict.

2026-07-09Accepted
calendar.rs computes NYSE open/close in America/New_York local time via chrono-tz and converts to UTC per date; regular_close honors early_closes.
Rationale
Stage 2/3 ingest a real two-year window that crosses the US DST boundary twice a year — a fixed-UTC calendar would misclassify session status for roughly half of any real dataset.
Consequences
Reworked Task 2, threaded early_closes through call sites, added DST-crossing and early-close tests.

Stage 0 plan · Task 2
2026-07-09Accepted
build_one sums contiguous price bars across signal_time→exit_time for horizons wider than one price_interval, and drops (not truncates) any horizon a gap prevents from being fully covered.
Rationale
The spec's experiment matrix needs horizons like "next four hours" over one-hour bars — the exact-single-bar-match design silently produced zero observations instead.
Consequences
Reworked Task 6; added a test proving a 240-minute horizon aggregates 4 bars mid-session and is dropped near close.

Stage 0 plan · Task 6
2026-07-09Accepted
entry_time == signal_time is correct and intentional — the spec's phrase "enter at the next tradable bar after signal_time" is imprecise, not the implementation.
Rationale
available_at <= signal_time == entry_time already guarantees no lookahead. A literal "next bar" reading would be an undocumented, unnecessary latency assumption.
Consequences
Spec left unedited (frozen/historical); this entry is the canonical wording. Added a dedicated no-lookahead test to Task 6.

Round 2 — adversarial workflow review · 2026-07-10
2026-07-10Accepted
run, analyze, and backtest accept an optional --run-id override (default: config.run_id) naming the runs/<run_id>/ directory for that invocation.
Rationale
run_id was a single static value, so every backtest --cost-bps <n> rerun overwrote the previous cost's reports — silently breaking the spec's Decision Demo "compare both runs" step.
Consequences
Tasks 1, 7, 8, 9 now thread an effective run_id; a new CLI test keeps separate reports per --run-id.
