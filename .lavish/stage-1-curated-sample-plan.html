Rev 3   “What does ‘curated-file adapter’ mean? Can we use simpler language?”

Fair — that was three pieces of jargon stacked up. It's gone.

I renamed it in the code and config too, not just the prose — news_source: "curated" would have been exactly as opaque sitting in a config file. It's now "saved_files". Here is the whole vocabulary in plain words:

TermIn plain words
source adapter A swappable way of getting the data. The rest of the pipeline doesn't know or care where the news and prices came from — it just asks a “source” for them. Swapping the source is the only thing that differs between Stage 0, 1, and 2; nothing downstream changes.
the fixture source Made-up data. Stage 0's synthetic articles and prices, generated by code. Fast, repeatable, completely fake.
the saved-file reader
(was: “curated-file adapter”)		 Real data, read from files on disk. This is what Stage 1 builds. A human fetches real payloads from Massive/GDELT/Alpaca once, saves them into the repo, and the reader loads them. No network calls when it runs. (The spec calls this “curated-file mode” — “curated” just means a human hand-picked the sample instead of a script scraping everything.)
the API source Real data, fetched live from the vendors. Stage 2's job. Needs API keys, rate limits, retries — everything Stage 1 deliberately avoids.
payload The raw response a vendor hands you — a blob of JSON or CSV, exactly as it arrived, before we've touched it.
quarantine “This article is broken; we can't trust it.” Bad timestamps, missing text.
exclusion “This article is fine, it's just not part of our sample.” Wrong dates, wrong symbols, or a duplicate. Deliberately not the same thing as quarantine — that distinction is the bug you caught.
Read this first. Stage 1's plan was written by auditing the implemented Stage 0 code against the spec — not by reading the spec alone. That audit found three structural gaps and one live, silent defect. Those findings, not the spec's one-sentence description of Stage 1, are what shaped this plan.
Rev 2   Changes from your review

What your annotations changed

  • “skipped? Or is quarantined the more precise word?” — you found a real bug in the design. You were right, and it was worse than a wording problem. Quarantine means “this row cannot be trusted”; an article outside the date window is perfectly trustworthy and merely out of sample. Since quarantine_rate drives the stop verdict, lumping them together would let a sample boundary masquerade as a data-quality failure and halt Stage 3 for no reason. Quarantine (quality) and exclusion (scope) are now separate dispositions with separate counters. → Decision 15 rewritten; new counter-test a_high_exclusion_rate_does_not_yield_stop.
  • “How are we going to get the real payloads then?” — answered below in Decision 12. Short version: a throwaway curl script, run once, outside the binary.
  • “Don't worry about ToS.” — open question S1-D closed and removed.
  • “What does ‘curated-file adapter’ mean? Can we use simpler language?” — Yes. See the Rev 3 glossary above; the jargon is renamed throughout the plan, the code, and the config.

What the code audit found

Live defect   src/backtest.rs:59–84 · src/sentiment.rs:12–29

Degenerate thresholds silently produce an all-long book

The Stage 0 sentiment lexicon is 14 words total, built for fixture headlines. Real headlines will miss it, so most articles score exactly 0.0. Both backtest thresholds are then derived from that distribution: 

let long_threshold  = quantile(sentiments, long_quantile);   // 0.8 -> 0.0
let short_threshold = quantile(sentiments, short_quantile);  // 0.2 -> 0.0

let side = if row.mean_sentiment >= long_threshold {  "long"  }   // 0.0 >= 0.0  ✓ ALWAYS
      else if row.mean_sentiment <= short_threshold { "short" }   // unreachable
      else { continue };

Both thresholds collapse to 0.0. Because the long branch tests >=, every neutral observation is classified long, the short branch never fires, and the run emits an all-long book with plausible-looking metrics. It fails silently, not loudly. The same tie-collapse makes analysis.rs::top_minus_bottom bucket on ties, so the verdict flips on noise.

A pipeline that cannot distinguish “no signal” from “no signal resolution” will mislead Stage 3. → Decision 17, Task 7.

Structural gap   src/pipeline.rs:57

The source-adapter seam the spec requires does not exist

The spec: “Vendor-specific code stays behind news and price source adapters. Fixture, curated-file, and API modes must produce the same normalized schemas.” But create_dataset_snapshot calls generate_fixture(config) directly. There is no trait, no seam, and no ingest subcommand (the spec's architecture lists one). → Decision 12, Task 1.

Structural gap   src/domain/article.rs:32

A missing timestamp is unrepresentable, so it cannot be quarantined

RawArticle.published_at is a non-optional DateTime<Utc>. The spec requires “quarantine articles with absent or ambiguous publication times” — and timestamp inspection is Stage 1's stated purpose. A schema that cannot represent a missing timestamp cannot quarantine one. The fixture never surfaced this because it always generates a clean timestamp. → Decision 15, Task 2.

Structural gap   src/pipeline.rs:126–127

The dataset manifest's date range is a hardcoded fixture literal

date_start: 2026-06-29, date_end: 2026-07-07 are written as literals. On real data that manifest is silently wrong — and the manifest is the lineage artifact every later stage and every reproducibility claim depends on. → Decision 16, Task 4.

The three decisions you made

Decision 12   Data intake

Read real data from saved files — no network code

Real payloads are committed to fixtures/saved_sample/. The code reads them off disk through a swappable data source (NewsSource/PriceSource). Stage 2 later slots the live-API source into the same place, without touching the research loop.

“How do we actually get the payloads, then?” — a throwaway curl script, scripts/fetch_sample.sh, run once, outside the Rust binary: 

# scripts/fetch_sample.sh   (NOT application code)
curl -H "Authorization: Bearer $MASSIVE_API_KEY" \
     "https://api.massive.com/v2/reference/news?ticker=SPY&..." \
     -o fixtures/saved_sample/massive_news_SPY.json
# ... GDELT export, Alpaca bars ...

The distinction that matters: no HTTP dependency enters Cargo.toml, and the pipeline still reads only from disk. The script keeps the sample reproducible and refreshable — you can re-run it — rather than leaving a pile of mystery files in the repo. API keys come from the environment and are never committed. Building fetching into the app would collapse Stage 2 into Stage 1 and force API keys, rate limits, and retries before a single real timestamp has been inspected by hand — which is Stage 1's entire purpose. Staying offline also means the sample cannot rot when a vendor changes an endpoint.

Decision 13   Sample scope

SPY / QQQ / DIA / IWM over one real trading week

Window
One real week containing a weekend boundary and a real holiday or early close
Volume
~140 hourly bars · ~50–150 real articles
Rationale
Small enough to eyeball every published_at; wide enough that observations, analyze, and backtest all produce non-empty output and calendar.rs meets real dates

Stage 1's backtest numbers are mechanically valid and statistically meaningless. Their only job is to prove real data flows through the loop. If a Stage 1 summary is ever quoted as signal evidence, that is a misuse.

Decision 14   Decision gate · supersedes half of Decision 6

Build the full 5-value vocabulary now; keep the baselines deferred

Decision 6 deferred stop/expand data/expand sources because “the fixture has no real data-quality signal to react to.” Stage 1 is the first stage that does — so these three values become verifiable here for the first time, and Stage 3's go/no-go gate arrives already exercised rather than written blind against the two-year dataset.

Data-quality gates are evaluated before signal gates — a signal result computed on unusable data is not a signal result:

if quarantine_rate    > max            -> "stop"             // timestamps unreliable
if degenerate || lexicon_hit_rate < min -> "expand sources"   // distribution too coarse to test
if source_set_coverage < min           -> "expand sources"
if articles_per_signal < min           -> "expand data"
if observed_spread > shuffled && > 0    -> "continue"
                                        -> "revise"

The 4 baselines (always-flat, random, prior-return momentum) stay deferred by the same logic inverted: baselines measure signal quality, and a 4-symbol / 5-day sample has no signal to measure. Decision 6's baseline half remains open and now gates Stage 2/3 alone.

Tasks

1

Make the data source swappable

NewsSource/PriceSource traits; move the fixture generator behind them unchanged; rename Stage0Config → PipelineConfig; add the spec's missing ingest subcommand. Pinning test: the fixture must still produce the same dataset_id.

2

Optional publication times · quarantine vs exclusion rev 2

published_at becomes Option on RawArticle, stays non-optional on NormalizedArticle — surviving quarantine is what makes an article normalized. Set-aside rows carry one of two dispositions: quarantined (quality: missing/unparseable/ambiguous timestamp) or excluded (scope: out of window, no relevant symbol, duplicate). Only quarantine feeds quarantine_rate. Invariant: every raw article lands in exactly one of normalized / quarantined / excluded.

3

The saved-file reader and the fetch script rev 2

scripts/fetch_sample.sh (throwaway curl, not app code) pulls the payloads once; they're committed. Vendor parsers for Massive JSON, GDELT CSV, Alpaca bars. fixtures/saved_sample/README.md records exact provenance per file. An empty curated_dir is a hard error, never a silent empty dataset.

4

Truthful dataset manifests

Derive date_start/date_end from the data. Articles outside the configured window are excluded — not quarantined — and are not allowed to silently widen the manifest. The configured window is an assertion, not a suggestion.

5

Real calendar and real bar alignment investigation

NYSE opens at 09:30 — not an hour boundary. If real Alpaca bars are clock-aligned, Decision 3's coverage-drop rule could silently discard most of the sample. Step 1 is to look at the payload, before writing any code.

6

Deduplication that survives real syndication investigation

Dedupe currently keys on url + title, which cannot catch a story republished under a different URL. Confirm the failure empirically first. If syndication doesn't appear in the sample, say so and stop — don't build machinery against a problem the data doesn't have.

7

Degenerate-distribution guard the defect

Zero trades and an expand sources verdict — never an all-long book. The required test must be shown to fail against the current backtest.rs before the fix; that's the proof the bug was real. A counter-test guards the check from being over-eager and suppressing a healthy Stage 3 distribution.

8

The full 5-value decision vocabulary

All thresholds live in config — no magic numbers in the verdict function, because Stage 3 will need to tune and audit them. The summary must report which metric tripped a non-continue verdict; an unauditable verdict is ceremony.

9

The leakage and timestamp audit report

The artifact a human actually reads: set_aside.csv (quarantined and excluded rows, grouped by disposition then reason, so a quality problem never hides inside a pile of scope exclusions) and timestamp_audit.csv (published_at vs available_at vs deferral, per article). Plus a machine-checked no-lookahead assertion over the real sample that fails the run on violation.

10

Demo, verification, and handoff

Fixture path must stay behavior-identical. Run the spec's Decision Demo on real data. Hand-trace one real article end to end — raw JSON → normalized → observation → trade — and write up what reality actually did to the pipeline.

Open questions — only the payloads can answer these

IDQuestionBlocksHow to resolve
S1-A Are real Alpaca hourly bars session-aligned (09:30 ET) or clock-aligned (09:00 ET)? Task 5 Look at the payload. Cannot be answered from documentation or reasoning. If clock-aligned, Decision 3's coverage-drop rule may discard most of the sample.
S1-B Does real syndication (same story, different URLs) actually appear in a 5-day / 4-ETF sample? Scope of Task 6, not Stage 1 Count it in the sample. A “no” is a valid answer that defers dedupe work to Stage 2.
S1-C Is the 14-word lexicon salvageable, or does the sentiment scorer need replacing? Stage 2, not Stage 1 Stage 1 measures lexicon_hit_rate and deliberately does not attempt the fix. If it's near zero, replacing the scorer becomes Stage 2's first design question.

S1-D (vendor terms for committing payloads) — closed at your direction. The payloads get committed.

Explicitly not in scope

  • Network code in the pipeline. No HTTP dependency in Cargo.toml, no rate limits, no retries, no API keys read by the binary. Payloads come from a one-shot curl script and are committed; the Rust code reads only from disk. Fetching becomes a real, tested part of the application in Stage 2.
  • The 4 baselines. They measure signal quality; this sample has no signal to measure. Decision 6's baseline half stays open.
  • Any claim about whether sentiment predicts returns. Stage 1's numbers are not evidence.
  • Rewriting the sentiment scorer. Stage 1 measures the lexicon's miss rate and refuses to trade on a degenerate distribution. It does not fix it.
