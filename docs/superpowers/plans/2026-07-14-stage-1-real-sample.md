**Goal:** Replace the fixture's synthetic inputs with a small, hand-saved sample of *real* Massive/GDELT/Alpaca payloads read from checked-in files, and use the resulting collision with reality to prove — by inspection and by test — that the pipeline's timestamp handling, leakage guarantees, deduplication, and decision gates survive contact with real data.

**Architecture:** Right now the pipeline has exactly one way of getting data, and it is hardwired: `pipeline.rs:57` calls `generate_fixture()` directly. Stage 1 makes the data source **swappable**, and adds the second of three sources.

The spec has always required this (*"Vendor-specific code stays behind news and price source adapters. Fixture, curated-file, and API modes must produce the same normalized schemas."*) — Stage 0 just never built it. Stage 1 adds `NewsSource`/`PriceSource` traits, moves the existing fixture generator behind them unchanged, and adds a **saved-file reader** that loads real vendor payloads from `fixtures/saved_sample/`. **No network code.** Stage 2 later adds the live-API source behind the same traits, without touching the research loop.

## Plain-language glossary

The spec and the Rust code both use some compressed vocabulary. This is what it means.

| Term | In plain words |
|---|---|
| **source adapter** | A swappable way of getting the data. The rest of the pipeline doesn't know or care where the news and prices came from — it just asks a "source" for them. Swapping the source is how Stage 0, 1, and 2 differ; nothing downstream changes. |
| **the fixture source** | *Made-up data.* Stage 0's synthetic articles and price bars, generated by code. Fast, deterministic, and completely fake. |
| **the saved-file reader** | *Real data, read from files on disk.* This is what Stage 1 builds. A human fetches real payloads from Massive/GDELT/Alpaca once, saves them into the repo, and the reader loads them. No network calls at run time. (The spec calls this "curated-file mode"; "curated" just means a human hand-picked the sample rather than a script scraping everything.) |
| **the API source** | *Real data, fetched live from the vendors.* Stage 2's job. Needs API keys, rate limits, retries — all the things Stage 1 deliberately avoids. |
| **payload** | The raw response a vendor gives you — a blob of JSON or CSV, exactly as it arrived, before we've touched it. |
| **quarantine** | "This article is broken; we can't trust it." Reserved for bad timestamps and missing text. |
| **exclusion** | "This article is fine, it's just not part of our sample." Out of date range, not about our symbols, or a duplicate. Deliberately **not** the same thing as quarantine — see Task 2. |

**Tech Stack:** unchanged from Stage 0 (Rust 2024, `clap`, `serde`, `serde_json`, `csv`, `chrono`, `chrono-tz`, `arrow`, `parquet`, `serde_arrow`, `plotters`, `sha2`, `hex`, `assert_cmd`, `tempfile`). No new runtime dependencies. Stage 1 deliberately adds **no HTTP client** — that is Stage 2's decision to make.

## Scope

This plan implements Stage 1 from `docs/superpowers/specs/2026-07-08-correlation-first-pipeline-design.md`: *"hand-picked real sample for timestamp and leakage inspection."*

Stage 1 must prove these properties before any API work (Stage 2):

- Real vendor payloads flow through the same normalize → observe → analyze → backtest → report loop as the fixture, producing identical schemas.
- Every article's `published_at` is human-checkable, and articles with absent or ambiguous publication times are **quarantined**, not silently dropped or defaulted.
- No observation can use an article whose `available_at` postdates its `entry_time` — asserted against real timestamps, not synthetic ones.
- The real trading calendar (a real weekend, a real holiday or early close, real Alpaca bar alignment) is handled correctly by `calendar.rs`.
- Real syndicated/republished articles are deduplicated.
- The pipeline **refuses to produce a verdict** when the sentiment distribution is degenerate, rather than silently emitting an all-long book.
- Reports emit the spec's full 5-value decision vocabulary, driven by real data-quality evidence.

### The sample (design.md Decision 13)

- **Symbols:** `SPY`, `QQQ`, `DIA`, `IWM`.
- **Window:** one real, recent trading week that contains a weekend boundary and at least one holiday **or** early close. Pick the window from the real NYSE calendar; record the exact dates in `configs/stage1_saved_sample.json`.
- **Expected volume:** ~140 hourly bars, ~50-150 real articles.

### Explicitly NOT in scope

- **Network code *in the pipeline*.** No `reqwest`, no HTTP dependency in `Cargo.toml`, no rate-limit handling, no retry/backoff, no API keys read by the binary. The payloads are fetched **once, by a throwaway `curl` script** (`scripts/fetch_sample.sh`, Task 3 Step 1) and committed; the Rust code reads only from disk. Fetching becomes a real, tested, rate-limit-aware part of the application in **Stage 2**.
- **The 4 baselines** (always-flat, random signal, prior-return momentum). Baselines measure *signal quality*, and a 4-symbol / 5-day sample cannot say anything meaningful about signal. Decision 6's baseline half stays open and still gates Stage 2/3. (design.md Decision 14.)
- **Any claim about whether sentiment predicts returns.** Stage 1's backtest output is mechanically valid and statistically meaningless. Its numbers must never be quoted as evidence.
- **Rewriting the sentiment scorer.** Stage 1 *measures* the lexicon's miss rate and refuses to trade on a degenerate distribution; it does not fix the lexicon. (Deliberate — see Open Question S1-C.)

## Task 1 — Make the data source swappable

**Why:** The spec requires vendor code to sit behind source adapters, and requires the fixture, saved-file, and live-API sources to all produce identical normalized schemas. But `pipeline.rs:57` calls `generate_fixture(config)` directly — the data source is hardwired, so there is nothing to swap. Every later stage depends on this seam existing.

- [ ] Step 1: Create `src/source/mod.rs` with the two traits. Keep them narrow — they return raw vendor records and nothing else. Normalization stays where it is.

```rust
pub trait NewsSource {
    /// Source names this adapter contributes, for `source_catalog.parquet`.
    fn sources(&self) -> Vec<SourceCatalogRow>;
    fn fetch_raw_articles(&self, config: &PipelineConfig) -> Result<Vec<RawArticle>>;
}

pub trait PriceSource {
    fn fetch_price_bars(&self, config: &PipelineConfig) -> Result<Vec<PriceBar>>;
}

pub fn news_source(config: &PipelineConfig) -> Result<Box<dyn NewsSource>> {
    match config.news_source.as_str() {
        "fixture" => Ok(Box::new(FixtureSource)),
        "saved_files" => Ok(Box::new(SavedFileSource::new(&config.saved_files_dir)?)),
        other => bail!("unknown news_source {other} (expected fixture|saved_files)"),
    }
}
```

- [ ] Step 2: Move `src/fixture.rs` → `src/source/fixture.rs`. Wrap `generate_fixture` in a `FixtureSource` implementing both traits. **Do not change its logic.** Stage 0's 45 tests must still pass unmodified except for import paths.
- [ ] Step 3: Rename `Stage0Config` → `PipelineConfig` throughout. Add fields: `news_source: String`, `price_source: String`, `saved_files_dir: Option<String>`.
- [ ] Step 4: `validate()` rejects an unknown source name, and rejects `news_source == "saved_files"` with no `saved_files_dir`.
- [ ] Step 5: Rewrite `create_dataset_snapshot` to obtain articles and bars from the adapters. Build `source_catalog` from `NewsSource::sources()` rather than by scanning articles.
- [ ] Step 6: Add the `ingest` CLI subcommand (spec architecture step 2 — Stage 0 never built it). It writes a dataset snapshot from the configured sources and prints `dataset_id=<id>`. `fixture` becomes a thin alias that forces `news_source=fixture`.

**Required tests:**
- `fixture_source_and_saved_file_source_produce_the_same_normalized_schema` — both adapters' output normalizes into `Vec<NormalizedArticle>` with identical field sets (the spec's explicit requirement).
- `unknown_source_name_is_a_config_error_not_a_panic`

## Task 2 — Optional publication times and the quarantine path

**Why:** `domain/article.rs:32` types `published_at` as a non-optional `DateTime<Utc>`. The spec requires *"Quarantine articles with absent or ambiguous publication times"*, and Stage 1's whole purpose is timestamp inspection. **A schema that cannot represent a missing timestamp cannot quarantine one.** The fixture never surfaced this because it always generates a clean timestamp. (design.md Decision 15.)

- [ ] Step 1: `RawArticle.published_at: Option<DateTime<Utc>>`. `NormalizedArticle.published_at` stays **non-optional** — a normalized article is by definition one that survived quarantine. This is the invariant the type system should carry.
- [ ] Step 2: Add to `domain/article.rs`:

```rust
/// A DATA-QUALITY failure: this article cannot be trusted.
/// Feeds `quarantine_rate`, which drives the `stop` verdict (Task 8).
#[derive(Debug, Clone, PartialEq, Eq, Serialize, Deserialize)]
#[serde(rename_all = "snake_case")]
pub enum QuarantineReason {
    MissingPublishedAt,
    UnparseablePublishedAt,
    AmbiguousPublishedAt,
    MissingTitleAndSummary,
}

/// A SCOPE exclusion: this article is fine, it just isn't part of this sample.
/// Counted and reported, but MUST NOT feed `quarantine_rate`.
#[derive(Debug, Clone, PartialEq, Eq, Serialize, Deserialize)]
#[serde(rename_all = "snake_case")]
pub enum ExclusionReason {
    OutsideDatasetWindow,
    NoRelevantSymbol,
    DuplicateOf { article_id: String },
}

pub struct SetAsideArticle {
    pub vendor_id: String,
    pub source: String,
    pub raw_published_at: Option<String>,  // the vendor's ORIGINAL string, unparsed
    pub title: String,
    pub url: String,
    pub disposition: Disposition,          // Quarantined(reason) | Excluded(reason)
}
```

**Quarantine and exclusion are different things and must not share a counter** (design.md Decision 15). Quarantine means *"we cannot trust this row"* — the spec's *"absent or ambiguous publication times."* Exclusion means *"this row is fine, it is simply not in our sample"* — an article outside the date window, or about a symbol we do not trade. Collapsing them corrupts the `stop` gate: `quarantine_rate` drives `stop` ("timestamps unreliable"), so counting a perfectly good out-of-window article as quarantined would make a **sample boundary** look like a **data-quality failure** and could halt Stage 3 for no reason.

Retain `raw_published_at` as the vendor's original string. The spec requires *"Normalize all timestamps to UTC while retaining original timestamps"*, and for a quarantined row the unparsed original **is** the evidence a human needs.

- [ ] Step 3: `normalize_articles` returns `(Vec<NormalizedArticle>, Vec<SetAsideArticle>)`. Nothing is silently dropped — every input article lands in exactly one of: normalized, quarantined, or excluded.
- [ ] Step 4: `article_id` currently hashes `(source, url, published_at)` (`normalize.rs:64-71`). With `published_at` now optional, quarantined articles have no stable id from that tuple. Hash `(source, url)` for the quarantine record and keep the 3-tuple for normalized articles.
- [ ] Step 5: Write `set_aside_articles.parquet` into the dataset snapshot and add it to the manifest's `files` list, with **separate** quarantined and excluded row counts.

**Required tests:**
- `article_with_no_published_at_is_quarantined_not_dropped`
- `article_with_unparseable_published_at_retains_its_original_vendor_string`
- `every_raw_article_lands_in_exactly_one_of_normalized_quarantined_or_excluded` — the no-silent-drop invariant.
- `an_out_of_window_article_is_excluded_and_does_not_raise_the_quarantine_rate` — guards the `stop` gate against a false positive caused by a sample boundary.
- `quarantined_and_excluded_row_counts_are_reported_separately_in_the_dataset_manifest`

## Task 3 — The saved-file reader and the real payloads

**Why:** This is the stage's actual deliverable — real bytes from real vendors, in the repo, reproducible offline forever.

- [ ] Step 1: Write `scripts/fetch_sample.sh` — a **throwaway shell script**, not application code. Plain `curl` calls that hit Massive `/v2/reference/news`, the GDELT export, and Alpaca 1-hour bars for the 4 symbols over the chosen week, writing raw responses straight into `fixtures/saved_sample/` with no transformation. API keys come from the environment (`MASSIVE_API_KEY`, `APCA_API_KEY_ID`/`APCA_API_SECRET_KEY`) and are **never** committed.

  This is how the payloads get fetched *without putting network code in the pipeline*: the fetcher is a script that runs once, outside the Rust binary. **No HTTP dependency enters `Cargo.toml`, and the pipeline itself still reads only from disk.** The script makes the sample reproducible and refreshable — you can re-run it — while keeping the offline guarantee that makes Stage 1 permanently replayable even if a vendor changes an endpoint. Stage 2 is where fetching becomes a real, tested, rate-limit-aware part of the application.

- [ ] Step 2: Run the script and commit the resulting payloads. Write `fixtures/saved_sample/README.md` recording, for each file: the exact request URL, the fetch timestamp, the vendor, and any hand-editing performed (there should be **none** — payloads are retained unchanged, per the spec's *"Ingest source payloads once and retain them unchanged"*). A human eyeballs the payloads before committing; that inspection is Stage 1's whole point.
- [ ] Step 3: `src/source/vendor/massive.rs` — parse the real payload shape into `RawArticle`. Massive gives `published_utc`, `title`, `description`, `publisher.name`, `tickers`, `article_url`. Map `description` → `summary`. `source_kind = Finance`. Missing/blank `published_utc` → `None`, which Task 2 then quarantines.
- [ ] Step 4: `src/source/vendor/gdelt.rs` — parse the GDELT export CSV. GDELT has no ticker tags, so every GDELT article is `SectorTheme` or `MacroMarket`, `source_kind = Broad`. GDELT timestamps are `YYYYMMDDHHMMSS` in UTC — parse explicitly and quarantine anything that fails.
- [ ] Step 5: `src/source/vendor/alpaca.rs` — parse the bars payload into `PriceBar`. **Do not assume bar alignment** — see Task 5.
- [ ] Step 6: `src/source/saved_files.rs` — a `SavedFileSource` that reads whichever of the three file kinds are present in `saved_files_dir`, dispatches to the vendor parsers, and concatenates. It must be a hard error if `saved_files_dir` is empty or missing, never a silent empty dataset.

## Task 4 — Truthful dataset manifests

**Why:** `pipeline.rs:126-127` hardcodes `date_start: 2026-06-29, date_end: 2026-07-07` as literals. On real data that manifest is **silently wrong**, and the manifest is the lineage artifact every later stage and every reproducibility claim depends on. (design.md Decision 16.)

## Task 5 — Real calendar and real bar alignment

**Why:** Stage 0's synthetic bars were generated to align perfectly with `signal_time`. Real Alpaca hourly bars are not guaranteed to. The NYSE regular session opens at 09:30 ET — **not** on an hour boundary — so an hourly bar series either starts at 09:30 (session-aligned) or 09:00/10:00 (clock-aligned) with a partial first bar. `observations.rs::build_one` sums *contiguous* bars from `signal_time` to `exit_time` and drops a horizon it cannot fully cover (Decision 3). If real bars are clock-aligned, that drop rule may silently discard most of the sample. **This is a question the code cannot answer — only the payload can.**

- [ ] Step 1: **Inspect the real payload first.** Before writing code, print the distinct `minute` values of every bar timestamp in the Alpaca sample. Record the finding in the plan and in `fixtures/saved_sample/README.md`.
- [ ] Step 2: Make `signal_time` generation align to the observed bar grid rather than assuming a clock-hour grid. If the session's first bar is 09:30 ET, signal times are 09:30, 10:30, … not 09:00, 10:00, ….
- [ ] Step 3: Fill `configs/stage1_saved_sample.json`'s `holidays` and `early_closes` from the **real** NYSE calendar for the chosen week.
- [ ] Step 4: Add an observation-drop counter to the coverage report. If `build_one` drops a horizon for lack of bar coverage, that must appear as a number a human can see, not as an absent row. (The spec: *"Record … dropped rows, and coverage gaps."*)

**Required tests:**
- `real_alpaca_bars_align_to_the_signal_time_grid_used_by_build_observations` — the test that would have caught a silent mass-drop.
- `a_real_holiday_or_early_close_in_the_sample_week_is_honored_by_regular_close`
- `dropped_horizon_coverage_gaps_appear_in_the_coverage_report`

## Task 6 — Deduplication that survives real syndication

**Why:** `normalize.rs:96-102` keys dedupe on `url + title`, lowercased. Real syndication republishes the *same story* under *different URLs* (Reuters → Yahoo Finance → MSN), so a URL-keyed dedupe will **not** catch it, and the spec explicitly requires *"Deduplicate syndicated and republished articles before aggregation."* An undetected syndicated duplicate inflates `article_count` and double-weights `mean_sentiment` — i.e. it corrupts the signal, not just the counts.

- [ ] Step 1: Confirm the failure empirically against the saved sample before changing anything. Count exact-URL duplicates vs. same-title-different-URL duplicates. **If real syndication does not appear in the sample, say so and stop** — do not build machinery against a problem the data does not have. Record the finding either way.
- [ ] Step 2: If syndication is present, key dedupe on a normalized title (lowercase, strip punctuation, collapse whitespace) plus a publication-time proximity bucket, keeping the **earliest** `published_at` among the duplicates — the earliest is the only one that is not lookahead.
- [ ] Step 3: Record the collapsed duplicates' count in the coverage report so dedupe is visible, not invisible.

**Required tests:**
- `syndicated_articles_with_different_urls_and_the_same_title_collapse_to_one` (uses real rows from the saved sample, cited by vendor_id in the test)
- `deduplication_keeps_the_earliest_published_at_of_a_duplicate_group`
- `dedupe_collapse_count_appears_in_the_coverage_report`

## Task 7 — Degenerate-distribution guard

**Why: this is the defect Stage 1 exists to find.** `sentiment.rs:12-29` is a 14-word lexicon built for fixture headlines. Real headlines will miss it, so most articles score exactly `0.0`. `backtest.rs:59-66` then computes both thresholds from that distribution, so `long_threshold == short_threshold == 0.0`; and because `backtest.rs:78` tests `mean_sentiment >= long_threshold`, **every neutral observation is classified `long`, the short branch never fires, and the run emits an all-long book with plausible-looking metrics.** Silent, not loud. The same collapse makes `analysis.rs::top_minus_bottom` bucket on ties, so `recommendation` flips on noise. (design.md Decision 17.)

- [ ] Step 1: Add `lexicon_hit_rate` to sentiment scoring — the fraction of scored articles containing at least one lexicon term. Thread it into a new `DataQualityMetrics` per configuration.
- [ ] Step 2: Define a configuration as **degenerate** when `long_threshold <= short_threshold` (the thresholds have collapsed or inverted), or when the share of observations at the modal sentiment value exceeds a configured `max_modal_share` (default `0.8`).
- [ ] Step 3: A degenerate configuration produces **zero trades** and is reported as such. It must not produce an all-long book. Backtest metrics for it carry an explicit `degenerate: true` flag rather than a silent zero.
- [ ] Step 4: `analysis.rs` marks a degenerate configuration's verdict as `expand sources` (Task 8) — the distribution is too coarse to test, which is a *data* problem, not a signal result.

**Required tests:**
- `a_sentiment_distribution_that_is_mostly_zero_produces_no_trades_rather_than_an_all_long_book` — construct the exact degenerate case: 10 observations, 9 at sentiment `0.0`. Assert `trade_count == 0` and `degenerate == true`. **Assert that this test fails against the current `backtest.rs` before the fix** — that is the proof the bug was real.
- `collapsed_long_and_short_thresholds_are_detected_as_degenerate`
- `the_real_saved_sample_reports_its_lexicon_hit_rate`
- `a_healthy_spread_of_sentiment_values_is_not_flagged_degenerate` — guards against the check being over-eager and suppressing Stage 3.

## Task 8 — The full 5-value decision vocabulary

**Why:** Decision 6 deferred `stop`/`expand data`/`expand sources` because *"the fixture has no real data-quality signal to react to."* Stage 1 is the first stage that **does** — and these three values are precisely the ones that only real data can falsify. Building them here means Stage 3's go/no-go gate arrives already exercised rather than written blind against the two-year dataset. (design.md Decision 14.)

- [ ] Step 1: Add `DataQualityMetrics` per configuration: `quarantine_rate`, `articles_per_signal`, `source_set_coverage` (share of configured sources actually contributing), `lexicon_hit_rate`, `dropped_horizon_rate`. **`quarantine_rate` counts data-quality failures only** — scope exclusions (out-of-window, no-relevant-symbol, duplicate) are reported separately and must never inflate it, or a sample boundary would trip the `stop` gate (design.md Decision 15).
- [ ] Step 2: Implement the verdict as an explicit, ordered decision function. Evaluate data-quality gates **before** signal gates — a signal result computed on unusable data is not a signal result.

```rust
pub fn verdict(q: &DataQualityMetrics, s: &SignalMetrics, t: &VerdictThresholds) -> &'static str {
    if q.quarantine_rate > t.max_quarantine_rate { return "stop"; }            // timestamps unreliable
    if q.degenerate || q.lexicon_hit_rate < t.min_lexicon_hit_rate { return "expand sources"; }
    if q.source_set_coverage < t.min_source_coverage { return "expand sources"; }
    if q.articles_per_signal < t.min_articles_per_signal { return "expand data"; }
    if s.observed_top_minus_bottom > s.shuffled_top_minus_bottom
        && s.observed_top_minus_bottom > 0.0 { return "continue"; }
    "revise"
}
```

- [ ] Step 3: Put every threshold in `PipelineConfig` under a `verdict_thresholds` object. **No magic numbers in the verdict function** — Stage 3's gates will need tuning, and a hardcoded constant is a gate nobody can adjust or audit.
- [ ] Step 4: `write_summary` gains a verdict column and, for any non-`continue` verdict, the **reason** — the specific metric that tripped and its value. A verdict a human cannot audit is ceremony.

**Required tests:**
- `a_high_quarantine_rate_yields_stop`
- `a_high_exclusion_rate_does_not_yield_stop` — the counter-test for Decision 15's split. A sample full of out-of-window articles is a *scope* fact, not a *quality* failure.
- `a_degenerate_sentiment_distribution_yields_expand_sources`
- `too_few_articles_per_signal_yields_expand_data`
- `data_quality_gates_are_evaluated_before_signal_gates` — construct a case with *both* bad data quality and a positive spread; assert the verdict is the data-quality one, not `continue`. This is the test that stops Stage 3 from reporting a false positive.
- `every_verdict_value_in_the_spec_is_reachable` — all 5 of `stop`/`revise`/`expand data`/`expand sources`/`continue`.
- `the_summary_reports_the_metric_that_tripped_a_non_continue_verdict`

## Implementation Notes

- **The fixture path is the regression suite.** Every task must leave `configs/stage0_fixture.json` producing byte-identical results. Tasks 1 and 4 are the risky ones; their pinning tests exist for this.
- **Expect the vendor payload shapes in Tasks 3-5 to differ from anything written here.** The code sketches above are illustrative. The real payloads are the source of truth — this is the same adapter caveat that Stage 0 hit with `serde_arrow` 0.14.2, and the correct response is to adapt the code and record the deviation, not to force the payload to match the plan.
- **Stage 1's backtest numbers are not evidence.** If a summary from this stage is ever quoted as signal, that is a misuse. The sample is 4 symbols over 5 days.
- **Tasks 5 Step 1 and 6 Step 1 are investigations, not implementations.** They may conclude "no change needed." That is a successful outcome and must be recorded, not worked around.
